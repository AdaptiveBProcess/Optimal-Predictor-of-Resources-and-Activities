\begin{abstract}
%% Version enviada
Business process optimization through simulation has advanced along two prominent yet largely independent lines: resource allocation, which decides who executes each activity, and prescriptive process monitoring, which decides what should happen next. Although both target overlapping operational objectives, their models and action spaces remain separate, limiting the flexibility available to any single optimization approach. This article presents a method that unifies both perspectives through deep reinforcement learning (DRL). The proposed method expands the action space by allowing the agent to jointly select for a given case the next activity and its executing resource at each decision point, enabling it to explore process configurations that neither approach can reach in isolation. We formalize this problem as a Markov Decision Process over a stochastic simulation environment learned from event logs, and define an architecture in which a DRL agent learns a policy that maximizes compliance with service-level agreements (SLAs), defined as cycle time thresholds. The agent operates under a masked policy guided by learned branching probabilities to preserve realistic process behavior despite the enlarged action space. This learned policy gives flexibility to improve the process whereas prevent greedy or unrealistic decisions. We validated the flexibility of policy learning on three real-life event logs against heuristic and learning-based baselines. The evaluation was performed using both SLA compliance metrics and a multi-perspective similarity framework covering control-flow, temporal, resource, and congestion dimensions. Results show that the joint optimization of activity routing and resource allocation increases SLA compliance while preserving realistic process behavior across all measured perspectives.

\section{Introduction}
\label{sec:introduction}

The execution of business processes involves two intertwined decisions at every step: which activity to perform next for a given case, and which resource should perform it. In practice, these decisions are typically separated, process execution follows a control-flow model discovered from historical data, while resource allocation is handled by a scheduling policy layered on top. This separation is convenient because it allows the simulation to remain faithful to the original control-flow, but strategically suboptimal: the best resource assignment for a fixed routing may not achieve the performance of a jointly selected activity-resource pair that routes the case differently from the outset.

Recent work in both resource allocation and prescriptive process monitoring has explored learning-based approaches to process optimization, though from largely separate angles. On the resource allocation side, DRL has emerged as a promising framework for learning assignment policies in BPS, with recent work demonstrating improvements over heuristic baselines in cycle time reduction \cite{Optimizing_Resource_Allocation_Policies, Learning_policies_drl}. These approaches, however, uniformly fix the control-flow and restrict the agent to selecting resources for activities determined by a separate routing model. On the prescriptive process monitoring side, approaches such as \cite{ppm_optimal_policy_temporal_data, ppm_resource_constraints} explore activity-level recommendations, but typically optimize case outcomes through domain-specific interventions rather than through a general joint decision policy over the full action space. Neither line of work addresses the joint selection of activity and resource as a unified decision.

This article proposes a unified decision framework in which a DRL agent jointly selects, at each decision point, the next activity and its executing resource as a single action $(a,r)$. The key challenge is that expanding the action space in this way risks producing policies that achieve improvements by exploiting the simulator, \emph{i.e.}, by following paths that are fast but inconsistent with real process behavior. Our central contribution is a \textbf{flexible masked policy} that resolves this challenge: the mask grants the agent meaningful \textbf{flexibility} to explore alternative process paths, while constraining its action space to a process-consistent subset derived from learned branching probabilities. We introduce \textbf{flexibility}; the capacity to dynamically delimit the action space in a way that is tunable between process conformance and operational performance; as the missing dimension in both lines of work. This subset is specified through three complementary filters: (i) top-$k$, which limits the number of candidate activities; (ii) top-$p$, which bounds the cumulative probability mass of selectable actions; and (iii) a minimum routing probability threshold $p_{\mathrm{end}}^{\min}$, which prevents the agent from terminating a case prematurely. Together, these mechanisms preserve realistic routing behavior without sacrificing the agent's exploration capacity.

The full contribution of this article is threefold. First, we formalize joint activity routing and resource allocation as a Markov Decision Process (MDP) over a stochastic BPS environment learned from event logs, defining a case-aware state representation that includes the case's positional prefix and branching probability vector alongside global resource utilization features. Second, we introduce the masked policy as an algorithmic mechanism for realistic flexible routing, and characterize its role in bounding the agent's exploration to process-consistent behaviors. Third, we ground the behavioral analysis in the similarity metrics of \citeauthor{framework_for_measuring_the_quality_of_business_process_simulation_models}~\cite{framework_for_measuring_the_quality_of_business_process_simulation_models}, covering control-flow, temporal, resource, and congestion dimensions, and complement these with a set of performance metrics; including SLA compliance rate, compliance improvement ratio, cycle time dispersion, and resource utilization balance ; designed to assess the agent's operational impact. Together, these two layers form an evaluation framework which assesses both whether the agent improves operational performance and whether it does so while preserving process realism. For evaluation purposes, the optimization objective is a cycle time threshold derived as a percentile of the original log's empirical distribution, to comply with an artificial service level of agreement (SLA). The agent architecture and masked policy are objective-agnostic and can be applied to other operational metrics depending on business needs.

We validate the approach on four real-life event logs of varying scale and domain against five baselines forming a progressive ablation from fully random decisions through heuristic and ML-driven routing to DRL-based resource allocation and finally to full joint DRL control, directly isolating the contribution of each component.

The remainder of the paper is structured as follows. Section~\ref{sec:related_work} reviews related work and positions the contribution. Section~\ref{sec:formalization} formalizes the problem as an MDP. Section~\ref{sec:approach} describes the approach with particular attention to the masked policy and state representation. Section~\ref{sec:evaluation} presents the evaluation design and preliminary validation. Section~\ref{sec:conclusion} concludes and outlines future directions.

\section{Process Optimization, Related Work and Research Gap}
\label{sec:related_work}

\subsection{Simulation Environments}
\label{subsec:related_work_simulation_env}

%Both lines of work depend on the quality of the underlying simulation environment. 
Business process simulation has evolved through three broad approaches: data-driven process simulation (DDPS), which discovers process components such as routing probabilities and resource calendars directly from event logs; deep learning-based simulation, which replaces hand-crafted components with neural models to capture temporal and behavioral patterns; and hybrid simulation, a more recent and promising direction that combines the interpretable structure of DDPS with the predictive accuracy of deep learning. The approaches presented in~\cite{camargo2023hybrid} and ~\cite{Runtime_integration_of_simulation_for_business_processes} are representative of this hybrid line, the latter introducing runtime integration of ML predictions, known as RIMS, enabling queue-aware time estimation and decision-point modeling. The simulation design adopted in this article draws conceptually from this hybrid paradigm, particularly its runtime ML integration and decision-point modeling, extended to support agent-driven activity and resource selection.

\subsection{Related Work}

Research on business process optimization through simulation has converged around two largely independent lines of work, each addressing a different dimension of the decision problem this article seeks to unify.

\subsubsection{Resource Allocation}
\label{subsec:related_work_resource_allocation}
The dominant approach in this line treats the control-flow as fixed and restricts the agent to assigning resources to activities as they become ready. Reinforcement learning methods have been applied to this problem with encouraging results. \citeauthor{Learning_policies_drl}~\cite{Learning_policies_drl} propose both a DRL policy and a score-based value approximation for resource assignment, outperforming heuristic baselines across several archetypal process structures. \citeauthor{Optimizing_Resource_Allocation_Policies}~\cite{Optimizing_Resource_Allocation_Policies} combine DRL with a hybrid simulation model to evaluate the approach on real-world processes, reporting improvements over heuristics in the majority of evaluation scenarios. \citeauthor{Resource_allocation_using_task_similarity}~\cite{Resource_allocation_using_task_similarity} apply reinforcement learning to reduce work list entropy by exploiting task similarity, optimizing cycle time through smarter resource-to-activity matching. Beyond RL, \citeauthor{genetic_resource_allocations}~\cite{genetic_resource_allocations} apply genetic algorithms to discover optimal resource configurations for what-if scenarios. Across these approaches, the action space is uniformly limited to resource selection: the routing of cases through the process is determined externally and held constant during optimization.

\subsubsection{Prescriptive Process Monitoring}
%\label{subsec:related_work_prpm}
%Prescriptive process monitoring 
(PrPM) approaches operate at the activity level but typically frame decisions as domain-specific interventions rather than general routing choices. \citeauthor{ppm_resource_constraints}~\cite{ppm_resource_constraints} learn intervention policies under resource constraints using DRL combined with conformal prediction, optimizing case outcomes rather than cycle time. \citeauthor{ppm_optimal_policy_temporal_data}~\cite{ppm_optimal_policy_temporal_data} construct an MDP over trace clusters and apply policy iteration to recommend optimal next actions, though the action space is defined over abstract state clusters rather than concrete activity--resource pairs. Neither line of work addresses the joint selection of activity and resource as a unified, general-purpose decision.

\subsection{Positioning and Research Gap}
\label{subsec:related_work_positioning}

The two lines of work reviewed above share a common limitation, though it manifests differently in each. Resource allocation approaches offer no flexibility over routing: the control-flow is fixed by the simulation model, and the agent's decisions are confined to resource assignment within that structure. Prescriptive process monitoring approaches, on the other hand, do reason about which activity should occur next, but without a principled mechanism to bound how far the recommended path may deviate from observed process behavior. In both cases, the trade-off between process realism and optimization potential is not tunable---it is either fully resolved in favor of the original control-flow, or left largely unconstrained.

We argue that \emph{flexibility}, understood as the capacity to dynamically delimit the action space in a way that is tunable between process conformance and operational improvement, is the missing dimension in both lines of work. A decision framework that incorporates flexibility as a first-class property can interpolate between conservative behavior that closely mirrors the original process and more exploratory behavior that pursues performance gains through alternative routing, with the degree of exploration governed by interpretable parameters.

To the best of our knowledge, no prior work in business process optimization has formalized this notion of flexibility as part of the decision framework, nor proposed a mechanism to \emph{implement} it within a learned policy. this article addresses that gap by introducing a flexible masked policy that makes the action-space boundaries explicit and tunable, grounding them in the process's empirical routing behavior.

\section{Problem Formalization}
\label{sec:formalization}

\subsection{Process Model and Decision Points}
\label{subsec:process_model_decision_points}

Let $L=\{\sigma_1,\sigma_2,\dots,\sigma_n\}$ be an event log, where each
trace $\sigma=\langle e_1,e_2,\dots,e_\ell\rangle$ is a temporally ordered
sequence of events sharing a common case identifier. Each event
$e=(c,a,t_s,t_e,r)$ records the case $c\in C$, the executed activity $a\in A$,
the start and end timestamps $t_s,t_e\in T$, and the executing resource
$r\in R$. A resource $r$ is associated with a subset $A_r\subseteq A$ of
activities it is qualified to perform.

A \emph{decision point} arises at branching points in the process, where the
current execution allows the selection from a given set of a valid continuation for a case $c$. 
At each such point, two interdependent choices must be made jointly: which activity
$a\in A$ to execute next, and which resource $r\in R$ to assign to it. Prior
work resolves these choices separately, fixing activity selection through a
routing model and delegating only resource assignment to the optimization
policy. this article treats them as a single joint action $\alpha=(a,r)$.

\subsection{Optimization Problem Formulation}
\label{subsec:mdp_formulation}

We formalize the optimization problem as a Markov Decision Process (MDP). 
An MDP is mathematical model that provides a principled way to model an environment
in which an agent learns by interaction: rather than prescribing a strategy, 
it defines the space of situations the agent may encounter, the actions available in each,
and the reward signal that encodes the operational objective, leaving the strategy
itself to be learned. This is particularly appropriate here because the
outcomes of resource and routing decisions are stochastic, depend on the
current process state, and cannot be evaluated without simulating their
downstream effects. Because the state captures a fixed-size snapshot of the
simulation rather than the full history of every active case, the MDP
approximation relies on the Markov property holding sufficiently in
practice, an assumption shared by prior DRL-based BPS
work~\cite{Optimizing_Resource_Allocation_Policies, Learning_policies_drl}.

An MDP $\mathcal{M}=(\mathcal{S},\mathcal{A},\mathcal{T},\mathcal{R})$
is defined by four components. The \textbf{states} describe all
possible configurations the agent may encounter, each summarizing the
properties of the environment relevant to decision-making. In our case, a
state captures the process at a given moment: which cases are active, how long
each has been running, what activity each is executing or waiting for, and
which resources are busy or available. The \textbf{actions} represent all
interactions the agent can perform with the objective of transitioning the
environment toward a more desirable configuration. Here, an action is the
joint selection of a next activity and an executing resource $\alpha=(a,r)$
for a given case, constrained as formalized above. The \textbf{transition
function} encodes the stochastic behavior of the environment: given a state
and an action, it determines the resulting next state. In our setting, this
reflects how the agent's routing and allocation decisions propagate through
the simulation; a particular choice of activity and resource may reduce
waiting time, free a bottleneck resource, or accelerate case completion,
but the exact outcome depends on the state of the broader process. Finally,
the \textbf{reward function} encodes the operational objective, quantifying
the desirability of each transition so the agent can connect actions to
outcomes. This is where the optimization goal is defined: one could reward
the agent for reaching specific activities, for reducing cost, or; as this
paper proposes; for completing cases within a service-level agreement, so
that the agent learns to maximize the proportion of cases whose cycle time
falls below a prescribed threshold $T$.

\subsection{Flexibility as a Problem Property}
\label{subsec:flexibility}

The MDP formulation above leaves one degree of freedom unspecified: the
definition of $\mathcal{F}_c$. Any choice of $\mathcal{F}_c$ yields a valid
instance of the MDP, but the choice determines where on the spectrum between
process conformance and operational performance the agent operates. At one
extreme, setting $\mathcal{F}_c$ to the single most probable activity
reproduces the original routing distribution exactly, leaving no room to
improve beyond the baseline. At the other, an unconstrained
$\mathcal{F}_c=A$ allows the agent to exploit simulation artifacts; following
paths that are fast within the simulator but structurally inconsistent with
real process behavior.

We define \textbf{flexibility} as the dynamic delimitation of the action space,
tunable to favor either process conformance or operational performance. We treat 
flexibility not as an implementation detail but as an explicit, named 
property of the framework: the dynamic delimitation of the action space, 
tunable to favor either process conformance or operational performance. Rather 
than fixing the boundary of $\mathcal{A}_c$ as a modeling assumption, 
flexibility makes that boundary a configurable parameter, allowing the 
framework to interpolate between conservative behavior that closely mirrors 
the original process and exploratory behavior that pursues performance gains 
through alternative routing. Section~\ref{sec:approach} \textit{implements} this 
definition through the flexible masked policy, which specifies 
$\mathcal{F}_c$ in terms of learned branching probabilities and three 
interpretable control parameters.

\subsection{Simulation Environment}
\label{subsec:simulation_environment}

The architecture distinguishes between two components with separate responsibilities. The \textbf{simulator} executes the business process faithfully: it advances the cases based on a control-flow model, predicts processing and waiting times, and maintains the state of all active cases. The \textbf{environment} wraps the simulator and mediates the interaction with the DRL agent; it is responsible for four specific functions: computing the flexible mask $\mathcal{F}_c$ at each decision point, translating the simulation state into a vector representation observable by the agent, translating the agent's output back into concrete activity and resource assignments that the simulator can execute, and retrieving the elapsed cycle time to compute the reward signal. This separation keeps the simulator self-contained and reusable; the environment layer can be modified; for example to change the reward function or the state representation; without altering the underlying simulation logic.

The simulator is built around a policy interface architecture that decouples each behavioral component from the simulation engine itself. Six interfaces define the simulator's behavior: the \textbf{routing policy} determines the next activity at each decision point; the \textbf{processing time policy} determines how long an activity takes to execute; the \textbf{waiting time policy} models delays before execution begins; the \textbf{arrival policy} controls how new cases enter the process; the \textbf{calendar policy} governs resource availability over time; and the \textbf{resource policy} determines which resource is assigned to a given activity. Any implementation conforming to a given interface can be substituted without modifying the simulation engine, making the architecture extensible by design.

The current implementations instantiate each interface as follows. Routing follows an XOR-only policy, meaning the process structure admits sequential branching without parallel splits; branching probabilities are estimated as conditional activity frequencies computed directly from the log. Processing and arrival times are sampled empirically per activity from the observed log distributions, preserving the full distributional shape including heavy tails and multimodality that parametric fits tend to distort. The calendar policy normalizes resource availability per weekday and determines availability using a percentile-based threshold. The resource policy assigns resources based on skill compatibility. The waiting time policy is not used in the current evaluation. At the decision point, the routing and resource interfaces are overridden by the DRL agent through the environment layer; rather than sampling the next activity from the branching distribution, the environment exposes the decision to the agent, using the branching probabilities to compute $\mathcal{F}_c$ as described in Section~\ref{subsec:mask}.

\subsection{State Representation}
\label{subsec:state_representation}

The state $s\in\mathbb{R}^{d}$ is a fixed-size vector observed by the agent at
each decision point and normalized to $[0,1]$. It concatenates three groups of
features.
\\

\paragraph{Global resource features ($3|R|$).}
\begin{itemize}[label=$\bullet$, itemsep=0pt, topsep=1pt]
  \item \textbf{Utilization:} $u_i = \mathrm{count}(r_i) / \mathrm{cap}(r_i)$.
  \item \textbf{Assignment:} $\eta_i = j/|A|$ if resource $r_i$ is currently executing activity $a_j$, and $0$ otherwise.
  \item \textbf{Queue pressure:} $q_i = \min\!\bigl(|\mathrm{queue}(r_i)|/\theta_q,\, 1\bigr)$.
\end{itemize}

\paragraph{Demand features ($|A|$).}
\begin{itemize}[label=$\bullet$, itemsep=0pt, topsep=1pt]
  \item \textbf{Pending demand:} $\kappa_j = \min\!\bigl(|\mathrm{pending}(a_j)|/\theta_q,\, 1\bigr)$.
\end{itemize}

\paragraph{Case-specific features ($|A|+5$).}
\begin{itemize}[label=$\bullet$, itemsep=0pt, topsep=1pt]
  \item \textbf{Last activity:} $\ell_c = j/|A|$, where $a_j$ is the last executed activity in case $c$ (and $\ell_c=0$ if undefined).
  \item \textbf{Trace length:} $\lambda_c = \min\!\bigl(|\sigma_c|/\theta_\lambda,\, 1\bigr)$.
  \item \textbf{SLA urgency:} $\phi_c = \min\!\bigl((t_{\mathrm{now}}-t_s(e_1))/T,\, 1\bigr)$.
  \item \textbf{Branching probabilities:} $b_{c,j} = P(a_j \mid a_{\mathrm{last}})$ for $j=1,\dots,|A|$.
  \item \textbf{Time:} $\tau_h$ (hour of day) and $\tau_d$ (day of week), both scaled to $[0,1]$.
\end{itemize}

The full state vector is
\begin{equation}
\label{eq:state_vector}
\begin{aligned}
s=\Bigl[&
\underbrace{u_1,\eta_1,q_1,\dots,u_{|R|},\eta_{|R|},q_{|R|}}_{\text{global }(3|R|)},\;
\underbrace{\kappa_1,\dots,\kappa_{|A|}}_{\text{demand }(|A|)},\\
&
\underbrace{\ell_c,\lambda_c,\phi_c,\; b_{c,1},\dots,b_{c,|A|},\; \tau_h,\tau_d}_{\text{case }(|A|+5)}
\Bigr],
\end{aligned}
\end{equation}
with dimension $d=3|R|+2|A|+5$. The branching probability components 
$b_{c,j}$ play a dual role: they inform the agent's routing decisions and 
are used directly to compute $\mathcal{F}_c$ in the masked policy. Resource 
availability is not enforced as a hard constraint at action selection time; 
rather, the utilization and queue pressure features provide the agent with 
sufficient information to learn availability-aware assignment through the 
reward signal. This allows the policy to capture nuanced trade-offs; for 
instance, waiting for a more suitable resource versus immediately assigning 
a less suitable one; while keeping the action space definition clean and 
the policy general.The constants $\theta_q = 10$ and $\theta_\lambda = 20$ are the normalization 
caps for queue-based features and trace length respectively; they are fixed 
empirically to cover the range observed across the event logs used in the 
evaluation, and values exceeding the cap are clamped to $1$.

\subsection{The Flexible Masked Policy}
\label{subsec:mask}

The flexible masked policy is the central algorithmic contribution of this article and the mechanism through which flexibility; as defined in Section~\ref{sec:formalization}; is \textit{implemented}. At each decision point, before the agent selects an action, the set of candidate activities is restricted to a process-consistent subset $\mathcal{F}_c\subseteq A$ derived from the branching probability vector $\mathbf{b}_c$. Three complementary filters define this subset.

The \textbf{top-$k$ filter} retains only the $k$ activities with the highest branching probability, limiting routing choices to the most process-consistent options. The \textbf{top-$p$ filter} retains the minimal set of activities whose cumulative probability mass reaches or exceeds $p$, ensuring that retained candidates collectively account for a sufficient proportion of observed routing behavior. The \textbf{minimum terminal probability threshold} $p_{\mathrm{end}}^{\min}$ applies exclusively to terminal activities; it enforces a lower bound on the probability of any selectable terminal activity, preventing the agent from closing a case prematurely before sufficient process progress has been made, We denote $\Omega\subseteq A$  as the set of terminal states.The three conditions are applied conjunctively: an activity mustsatisfy all applicable filters simultaneously to be admitted into $\mathcal{F}_c$.


\[
\mathcal{F}_c
=
\Bigl\{a_j\in A \,\Big|\,
a_j\in \mathrm{top}\text{-}k(\mathbf{b}_c),\;
a_j\in \mathrm{top}\text{-}p(\mathbf{b}_c),\;
a_j \notin \Omega
\;\lor\;
P(a_j\mid a_{\text{last}})\ge p_{\mathrm{end}}^{\min}
\Bigr\}.
\]


Actions outside $\mathcal{A}_c$ receive a logit value of $-\infty$ prior to the softmax implicit in the categorical distribution from which the action is sampled, following standard masking practice in DRL~\cite{Learning_policies_drl}. The parameters $k$, $p$, and $p_{\mathrm{end}}^{\min}$ together control the degree of flexibility: lower values produce a conservative policy that closely mirrors the original routing distribution; higher values expand the agent's freedom to explore alternative paths at the cost of greater divergence from observed behavior.

\subsection{Agent Architecture}
\label{subsec:agent_architecture}

The agent is an Actor--Critic network trained with Proximal Policy Optimization (PPO). The actor outputs two probability distributions over the constrained action space: one over candidate activities $\mathcal{F}_c$ and one over qualified resources $R_{\text{qual}}(a)=\{r\in R : a\in A_r\}$, from which the joint action $\alpha=(a,r)$ is sampled. The critic takes the same state $s$ as input and outputs a scalar estimate $V(s)$ of the expected future return, used to compute advantages via Generalized Advantage Estimation (GAE) with parameters $\gamma$ and $\lambda$. Both actor and critic share a common trunk of fully connected layers with separate output heads.

Note that the discount factor $\gamma$ appears here as a hyperparameter of GAE rather than as a structural property of the MDP, since the true optimization objective, cycle time compliance, is an undiscounted, episode-level quantity. The PPO clipping objective prevents excessively large policy updates, which is particularly important in this setting where the reward signal is sparse at the episode level and the simulation is non-stationary due to concurrent case arrivals.

\subsection{Reward Function}
\label{subsec:reward}

The reward function guides the agent toward cycle time compliance through a
piecewise-linear design with two regimes: an \textbf{intermediate shaping
signal} issued at every step, and a \textbf{terminal gate} that encodes SLA
compliance at case completion. Let $ct(\sigma)$ denote the elapsed cycle time
of case $\sigma$ and $T$ the service-level threshold.

During execution, the shaping signal provides dense directional feedback:

\[
\mathcal{R}_{\text{int}}(s,\alpha)=
\begin{cases}
\dfrac{K}{100}\!\left(1-\dfrac{ct(\sigma)}{T}\right), &
  \text{if } ct(\sigma)<T,\\[10pt]
-\dfrac{K}{100}\cdot\dfrac{ct(\sigma)}{T}, &
  \text{if } ct(\sigma)\ge T.
\end{cases}
\]

When the elapsed time remains within the SLA budget, the agent receives a
positive reward that decreases linearly from $+K/100$ (at case start) to $0$
as $ct(\sigma)\to T$, continuously favoring faster progress. Once the threshold
is exceeded, the signal turns negative and grows in magnitude proportionally to
the degree of violation, increasing pressure to avoid further delay. At case
completion, the terminal reward overrides the shaping signal:

\[
\mathcal{R}_{\text{term}}(s,\alpha)=
\begin{cases}
+K, & ct(\sigma)<T,\\
-K, & ct(\sigma)\ge T.
\end{cases}
\]

The $K/100$ scaling factor on the intermediate signal is deliberate: it ensures
that the terminal reward dominates the learning objective, so the agent
optimizes for SLA compliance as an outcome rather than for step-level speed.
The intermediate component acts as a shaping term in the sense of
reward shaping~\cite{sutton_2018_rl_intro}, providing denser gradient
information without distorting the compliance objective. In PPO, these rewards
are incorporated into advantage estimates via GAE, improving convergence in the
long-horizon setting of business process execution.

\section{Evaluation}
\label{sec:evaluation}

The evaluation assesses two complementary properties of the proposed 
framework: whether the DRL agent with the flexible masked policy 
improves SLA compliance relative to the original process, and whether 
those improvements are achieved while preserving realistic process 
behavior across multiple perspectives. We evaluate the full DRL-AR 
policy on three real-life event logs and report both operational 
performance and behavioral similarity results.

meaningful finding.

The results reveal a consistent and interpretable pattern across 
all three logs. NGD distances ($0.41$--$0.47$) are substantially 
higher than reference values, which is the expected signature of 
a policy that exercises flexibility: the agent deviates from the 
original routing distribution because doing so is necessary to 
improve SLA compliance, and the degree of deviation is directly 
governed by the top-$k$ and top-$p$ parameters. This is the 
deliberate cost of flexibility rather than a simulator artifact.

By contrast, CED and CWD distances remain close to or within 
reference range across all three logs. The temporal seasonality 
and circadian workforce patterns of the original process are 
preserved, confirming that the agent respects calendar and 
resource availability structures even while rerouting cases. 
This is a substantive finding: the policy learns to improve 
performance within the temporal constraints of the process 
rather than by concentrating work in unrealistic periods.

RED and CTD distances are elevated relative to reference values, 
consistent with the agent accelerating case execution to meet 
SLA targets. Cases that complete faster necessarily exhibit 
a different internal temporal structure and a shifted cycle 
time distribution; this is not behavioral drift but the 
expected consequence of a policy that successfully compresses 
cycle times. The CAR distances are large for BPIC17, reflecting 
differences in the arrival process between training and 
evaluation periods; since the agent does not control case 
arrivals, this deviation is attributable to the arrival model 
rather than to the policy itself.

Taken together, the similarity profile confirms that the 
DRL-AR policy concentrates its behavioral changes in the 
dimensions it controls --- control-flow routing and case 
duration --- while leaving the temporal and workforce 
patterns of the original process substantially intact.

\subsection{Threats to Validity}
\label{subsec:threats}

Regarding internal validity, the current simulator supports 
XOR-only routing; processes with parallel activity structures 
are outside the present evaluation scope. Hyperparameter 
sensitivity of the flexibility parameters $k$, $p$, and 
$p_{\mathrm{end}}^{\min}$ is a further internal threat; 
configurations were selected empirically per log and a 
principled selection criterion remains future work. The absence 
of baseline comparisons (RA-RR, GP-RR, DM-RR, DM-DRL) in the 
current evaluation limits the ability to attribute compliance 
improvements specifically to joint optimization versus routing 
or resource allocation in isolation; the full ablation 
protocol described in the experimental design constitutes 
the natural next step. Regarding external validity, the 
evaluation covers three logs from two domains and results 
may not generalize to processes with substantially different 
structural characteristics. Regarding construct validity, 
the Chapela-Campa et al.\ similarity framework has been 
independently validated but may not capture all dimensions 
of behavioral fidelity relevant to specific application domains.

\section{Conclusion and Future Work}
\label{sec:conclusion}

This paper introduced a unified decision framework in which a DRL agent 
jointly selects the next activity and its executing resource as a single 
action $(a,r)$ at each decision point in a business process simulation. 
The central contribution is the formalization of \textbf{flexibility}; 
the dynamic delimitation of the action space, tunable between process 
conformance and operational performance; and its operationalization 
through the flexible masked policy, which grounds the action space 
boundary in learned branching probabilities and three interpretable 
parameters.

The evaluation demonstrates that the framework achieves its intended 
purpose on both dimensions. From a performance perspective, the DRL-AR 
agent consistently increases SLA compliance across three real-life logs 
and multiple threshold levels, with gains reaching up to $+30\%$ CIR 
at $T_{75}$ and, in the best cases, $+67.8\%$ at $T_{50}$ for BPIC17; 
more conservative improvements around $+5\%$ are consistently observed 
at $T_{95}$ across all logs. These results confirm that the agent 
genuinely restructures process execution rather than exploiting 
simulation artifacts. From a similarity perspective, the flexible masked 
policy demonstrably limits behavioral deviation: while NGD distances 
reflect the routing changes the agent deliberately introduces, circadian 
and workforce similarity metrics (CED, CWD) remain within the range of 
calibrated BPS models, confirming that the policy respects the temporal 
and resource patterns of the original process.

\textbf{Furture Work}: Several directions extend the present work naturally. A first priority 
is policy interpretability: introducing models such as decision trees or 
rule extraction techniques to characterize the agent's learned 
behavior, and comparing the inferred rules against the original process 
to identify which structural patterns the agent departs from and why.

A second direction concerns the shape of flexibility. Replacing the 
current hard top-$k$ and top-$p$ filters with a KL-divergence 
regularization term in the reward signal would allow the agent to 
continuously trade off between optimality and distributional similarity 
to the as-is process, drawing on techniques from variational 
approaches used in generative modeling.

Third, integrating case-aware ML models for branching probability 
estimation; such as LSTMs trained on activity prefixes; would provide 
richer routing signals that reflect each case's history, potentially 
enhancing SLA urgency estimation and enabling the agent to exploit 
case-level patterns the current state representation cannot capture.

Fourth, extending the simulator to support parallel branching through 
full Petri net integration would allow the framework to be applied to 
processes with AND-split structures, raising new questions about how 
the agent handles intra-case concurrency and resource competition.

Finally, the objective-agnostic architecture opens the door to 
alternative reward functions beyond SLA compliance, enabling what-if 
exploration grounded in operational goals such as outcome-based 
routing or cost minimization.
